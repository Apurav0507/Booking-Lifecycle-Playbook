Data layers — what lives where
Identity (write once)
cf_booking_id cf_query_code
Set at creation. Never change. These are the join keys between both systems.
Layer 2 — Operational snapshot
13 fields kept in sync via events. Drive routing, views, SLAs. Read-only in Zoho UI.
Layer 3 — Live widget
Full traveler details, room breakdown, payment ledger. Fetched live from Aryabhatta API on ticket open. Never stored in Zoho.
Layer 4 — Reconciliation
Nightly job compares Layer 2 fields across both systems. Heals drift. Surfaces sync health as daily metric.
Ownership rule — the most important principle
If a field originates in Aryabhatta — booking stage, payment status, supplier, dates — nobody updates it in Zoho. It flows in through events only. If a field originates in Zoho — agent notes, escalation tags, follow-up dates — nobody puts it in Aryabhatta. There is no shared ownership.

13 Zoho custom fields + 1 live layer · field-scoped updates only — never send full payload
Field-scoped update rule: The middleware never sends a full ticket payload to Zoho. Every event patches only the 2–3 fields it owns. This means an Aryabhatta update of cf_payment_status and a Zoho agent saving a note can happen simultaneously without one overwriting the other.
